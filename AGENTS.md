# Repository Guidelines

## Agent Operating Scope

This file is guidance for AI coding agents working in this repository. Keep changes small, specific, and aligned with the existing Dart package structure. Prefer reading nearby code before introducing new patterns, and do not rewrite generated or unrelated files unless the task requires it.

## Project Structure

This is a Dart client for the Revan API. Public exports start at `lib/revan_client.dart`; implementation lives in `lib/src/`. Shared HTTP and error handling code is in `lib/src/core/`. Trubar endpoint logic and DTOs are in `lib/src/trubar/`. Tests live in `test/`, and runnable examples belong in `example/`.

Model files generated by Freezed and JSON serialization sit beside their source files as `*.freezed.dart` and `*.g.dart`.

## Required Commands

Use FVM for all Dart commands:

- `fvm dart pub get` installs dependencies.
- `fvm dart analyze` runs static analysis.
- `fvm dart test` runs tests.
- `fvm dart format lib test example` formats Dart files.
- `fvm dart run build_runner build --delete-conflicting-outputs` regenerates model code.

Run `fvm dart analyze` and `fvm dart test` after behavior changes when practical.

## Coding Rules

Follow `package:lints/recommended.yaml` and standard Dart formatting. Use two-space indentation. File names use lowercase with underscores, for example `revan_api_client.dart`. Classes and enums use `UpperCamelCase`; methods, fields, and local variables use `lowerCamelCase`; private members start with `_`.

Do not manually edit `*.freezed.dart` or `*.g.dart`. Change the source model and regenerate.

## Testing Rules

Tests use `package:test`; Mockito is available for mocks. Put tests in `test/` and name files `*_test.dart`. Use `group()` for related behavior and write clear names such as `should initialize RevanApiTrubarEndpoint`.

Avoid tests that require a live backend. Use mocks or local fakes for API behavior.

## Commit Rules

Use Conventional Commits: `type(scope): summary`. Common types are `feat`, `fix`, `test`, `docs`, `refactor`, `ci`, and `chore`. Examples: `feat(trubar): add word data endpoint`, `test(api): cover client initialization`.

## Release Process

For pre-1.0 package versions, interpret semver with the public API significance shifted down one slot: `0.2.0` indicates a breaking change, `0.1.3` indicates a new feature, and `0.1.2+1` indicates a change that does not affect the public API. Do not use `+N` build metadata for feature releases.

The GitHub publish workflow is configured in `.github/workflows/publish.yml` and runs after the `Analyze & Test` workflow succeeds on `main`. Release prep should update `pubspec.yaml` and `CHANGELOG.md`, then validate with `fvm dart analyze` and `fvm dart test`.

## Safety Notes

Do not commit real API URLs, credentials, or tokens. Use placeholders such as `https://api.example.com`. Preserve user changes already present in the working tree.
